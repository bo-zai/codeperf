# CodePerf 架构总览（00 · 主设计文档）

> 本文件记录当前项目的真实定位、模块划分、检测边界与后续演进方向。编码若与本文档不一致，必须及时回写。

## 1. 工具定位

CodePerf 是面向 Java 项目的**提交前循环 I/O 放大风险检测工具**。第一阶段优先解决一类线上事故：开发在原有功能中加入循环，循环体内调用数据库、HTTP、RPC、SDK 或缓存访问，测试环境数据量较小未暴露问题，生产规模放大后导致接口响应显著变慢。

当前核心能力分为两层：

- 静态检测：CLI 在开发者本地 Git 流程中执行，对源码做 AST 分析，提前拦截循环内 I/O 放大风险。
- 动态检测：agent 仅负责在测试、预发或手工验证环境中打桩并上报运行证据，不再作为生产 attach 或本地 attach 的核心方案。

## 2. 关键设计约束

| 约束 | 当前结论 | 影响 |
|---|---|---|
| 生产安全 | 不在生产环境 attach，不采集生产明细数据 | 动态 agent 只能作为辅助证据，不能宣称生产实测 |
| Git 流程 | CLI 运行在本地 pre-commit/pre-push 或手工 scan | 必须输出简洁、可定位、可快速修复的结果 |
| 静态主规则 | `LoopIoAmplificationRule` 是第一阶段核心 | 优先识别循环内 DB/HTTP/RPC/SDK/Redis 等 I/O |
| 企业项目 | 支持多模块仓库 | CLI 与 HTML 报告必须按模块、文件、问题分层展示 |
| JDK 版本 | 项目整体保持 Java 8 兼容 | 禁止使用 Java 9+ API |
| 数据库范围 | 公司内部主要为 MySQL、MongoDB、Redis、GaussDB | I/O 分类与规则文案按这些技术栈优先设计 |

## 3. 模块划分

```text
codeperf-parent
├── codeperf-cli           提交前静态扫描、Git 归因、本地报告、结果上报
├── codeperf-agent         运行态打桩与采集，仅用于测试/预发/手工验证证据增强
├── codeperf-server        接收扫描任务、仓库、提交、规则、发现与通知相关数据
├── codeperf-demo-admin    多模块 demo：管理端场景，包含真实业务写法风险样例
├── codeperf-demo-app      多模块 demo：应用端场景，包含真实业务写法风险样例
└── codeperf-demo-common   多模块 demo：公共能力与跨模块调用风险样例
```

旧的单模块 `codeperf-demo` 已废弃，当前 demo 使用 `admin/app/common` 三模块结构模拟企业项目。

## 4. CLI 职责

`codeperf-cli` 是开发者进入测试前的第一道本地门禁，承担以下职责：

- 读取 `.codeperf.yml`，识别项目、源码根、模块配置、门禁阈值与报告路径。
- 基于 Git 计算本次变更范围，并在 `--all` 模式下扫描完整源码。
- 使用 AST 规则识别循环内 I/O 放大风险，支持一定深度的跨方法调用链分析。
- 做 Git blame 归因，区分本次变更风险、历史风险与未归因风险。
- 生成简洁控制台摘要和本地 JSON/HTML 报告。
- 在配置开启时，将静态扫描结果上报到 `codeperf-server`。

CLI 不负责启动被测应用，不负责注入 agent 参数，也不通过 attach 修改目标 JVM。

## 5. Agent 职责

`codeperf-agent` 的定位是运行态证据增强，不是生产检测核心。真实使用中由应用启动脚本或流水线环境预先配置：

```bash
-javaagent:/opt/codeperf/codeperf-agent.jar=/opt/app/.codeperf/agent.yml
```

agent 配置固定在配置文件中，CLI 不生成 `agent.yml`，也不在 doctor 中检查 agent 配置。动态数据需要通过仓库、提交、任务等稳定标识与静态扫描结果在 server 侧关联。

## 6. 报告分层

报告结论必须区分三类来源，避免过度承诺：

- 静态结构风险：源码结构上存在循环内 I/O 调用，适合提交前阻断。
- 测试/预发运行证据：agent 在非生产环境采集到对应调用或耗时证据。
- 生产规模推断风险：通过脱敏聚合指标、APM 指标或 DBA 统计推断生产规模下的放大风险。

本地 HTML 报告面向开发者修复，优先展示模块、文件、行号、调用链、证据与修复建议，而不是管理层大屏。

## 7. 当前优先级

第一阶段聚焦循环 I/O 放大风险，不把 CPU、内存、慢 SQL 作为核心门禁。后续可以在同一规则扩展接口下接入更多规则，但必须保持静态主流程、动态证据与生产规模画像三者边界清晰。
