# 生产规模画像

## 1. 目标

生产规模画像用于回答：

```text
这段循环在生产环境通常会执行多少次？
```

它不采集生产运行时明细，也不包含用户、订单、金额、手机号、地址等业务数据。

画像只包含脱敏聚合指标。

## 2. 为什么需要画像

静态规则能发现“循环内 I/O”，但不知道循环规模。

动态测试能发现“本次执行了多少次”，但测试数据可能远小于生产。

生产规模画像补上第三块证据：

```text
生产 P95/P99 下，这个循环可能被放大到什么程度？
```

## 3. 可接受的数据类型

允许输入：

- 总量：用户数、订单数、商品数、部门数、租户数。
- 分布：每个用户订单数 P50/P95/P99/max。
- 请求规模：某接口一次处理列表长度 P50/P95/P99。
- 命中规模：某查询条件在生产中的返回行数分布。
- 租户规模：大租户、大客户、大商户的数据量分布。

不允许输入：

- 生产明细行。
- 用户标识、手机号、地址、证件号。
- 订单明细、金额明细、支付信息。
- 原始请求参数和响应体。

## 4. 获取方式

### 4.1 DBA 聚合 SQL

由 DBA 在生产执行聚合 SQL，只导出统计结果。

示例：

```sql
SELECT COUNT(*) AS user_count FROM users;
```

按用户统计订单数分布：

```sql
SELECT user_id, COUNT(*) AS cnt
FROM orders
GROUP BY user_id;
```

导出前应再做聚合分位数，只保留 P50/P95/P99/max。

### 4.2 日志或网关指标

如果接口日志中已有列表长度、返回条数或业务范围，可以从日志平台聚合：

```text
entry = POST /api/orders/report
items_count P50 = 20
items_count P95 = 2000
items_count P99 = 8000
```

### 4.3 APM 或监控指标

如果已有 APM，可从接口维度提取：

- 请求量。
- P95/P99 响应时间。
- 下游调用次数分布。
- DB 查询次数分布。

注意：只取聚合指标，不导出 trace 明细。

### 4.4 手工业务估算

如果短期无法接入数据平台，可以由业务、测试、DBA 共同提供保守估算。

示例：

```yaml
entries:
  "POST /api/orders/report":
    loopInputs:
      users:
        prodP50: 100
        prodP95: 3000
        prodP99: 12000
        max: 50000
```

该方式精度较低，但比完全没有生产规模依据更好。

## 5. 建议文件形态

后续可新增 `perf-profile.yml`：

```yaml
version: 1
entries:
  "POST /api/orders/report":
    loopInputs:
      users:
        testP95: 50
        prodP50: 200
        prodP95: 3000
        prodP99: 12000
    notes:
      owner: order-team
      source: dba-aggregate-2026-07
```

规则报告引用该画像，不直接读取生产数据库。

## 6. 风险评估方式

基础估算：

```text
放大倍数 = 生产 P95 循环规模 / 测试循环规模
```

如果测试循环规模未知：

```text
使用生产 P95 绝对规模判断风险。
```

示例：

```text
测试执行 5 次
生产 P95 预计 2000 次
放大倍数 = 400
```

报告应提示：

```text
该风险在测试环境未暴露，不能作为生产安全依据。
```

## 7. 安全边界

生产规模画像必须遵守：

- 只存储聚合指标。
- 不包含可识别个人或业务对象的明细。
- 不自动连接生产数据库。
- 不要求 agent 进入生产 JVM。
- 指标来源和生成时间应记录在文档或 YAML 中，便于评审追溯。
