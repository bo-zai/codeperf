# 循环 I/O 放大事故模型

## 1. 事故模式

本项目要优先防止的事故不是普通慢接口，而是“循环内外部调用被生产数据量放大”的事故。

典型代码形态：

```java
for (Item item : items) {
    remoteClient.query(item.getId());
}
```

当 `items` 在测试环境只有少量数据时，接口可能表现正常；当生产环境 `items` 达到数百、数千甚至更高时，远程调用次数随循环次数线性增长，接口延迟被放大。

## 2. 事故链路

完整事故链路如下：

```text
需求开发
-> 在已有功能中加入循环
-> 循环体内调用外部接口/DB/RPC
-> 测试环境数据规模小
-> 测试未发现接口明显变慢
-> 生产数据规模大
-> 单次请求触发大量外部 I/O
-> 接口响应慢、线程堆积、下游服务压力升高
```

这类问题的关键不在于单次外部调用慢，而在于调用次数会被业务集合规模放大。

## 3. 为什么测试环境容易漏掉

测试环境通常存在以下差异：

- 数据量小，例如测试只覆盖 5 条记录，生产 P95 是 2000 条。
- 租户规模小，无法覆盖大客户、大部门、大商户场景。
- 测试用例只验证功能正确性，不验证大规模数据下的调用次数。
- 外部接口在测试环境响应更快，掩盖循环调用次数问题。
- 测试没有采集“循环次数、远程调用次数、同类 SQL 次数”等指标。

因此，测试环境接口不慢不能证明生产安全。

## 4. 风险本质

风险可以抽象成：

```text
总耗时 = 循环次数 N * 单次 I/O 耗时 T + 其他处理时间
```

如果 `N` 在生产环境远大于测试环境，即使 `T` 很小，总耗时也可能不可接受。

示例：

| 场景 | 循环次数 | 单次外部调用耗时 | 估算总耗时 |
|---|---:|---:|---:|
| 测试环境 | 5 | 20ms | 100ms |
| 生产 P95 | 2000 | 20ms | 40000ms |
| 生产 P99 | 8000 | 20ms | 160000ms |

这个估算不要求精确，但足以判断上线风险。

## 5. 应检测的 I/O 类型

第一阶段规则应覆盖以下调用：

- 循环内数据库查询或更新。
- 循环内 HTTP 调用。
- 循环内 RPC 调用。
- 循环内调用外部 SDK。
- 循环内调用看似本地 Service，但该 Service 内部继续调用外部 I/O。

其中，最后一种最难，需要后续通过调用图增强。

## 6. 修复方向

报告不只提示风险，也应给出修复方向：

- 改成批量接口，例如 `queryBatch(ids)`。
- 循环前预取数据，用 Map 在内存中关联。
- 对重复查询结果加缓存。
- 将同步逐条调用改成受限并发，但必须配合超时、限流和熔断。
- 将非关键路径改成异步处理。
- 如果必须逐条调用，要求给出生产规模证明和耗时预算。
