# CodePerf 项目重新定位

## 1. 背景

CodePerf 最初按“提交前性能检测工具”推进，当前实现已经覆盖静态扫描、动态 attach、agent 采集、报告和门禁。经过线上事故复盘后，项目目标需要收敛。

实际要解决的核心事故是：

```text
开发在已有功能中加入循环
循环体内调用外部接口或其他 I/O
测试环境数据量小，未暴露响应变慢
生产数据量大，上线后接口响应时间被线性放大
```

因此，CodePerf 第一阶段不应继续定位为“大而全的性能检测工具”，而应定位为：

```text
上线前循环 I/O 放大风险检测工具
```

## 2. 新定位

CodePerf 的第一阶段目标是：在代码合入前，识别本次变更或目标代码中是否存在“循环内调用外部 I/O”的结构性风险，并结合生产规模画像评估线上放大后果。

这里的外部 I/O 包括：

- 数据库访问：Repository、DAO、Mapper、JDBC、ORM 查询。
- HTTP 调用：Feign、RestTemplate、WebClient、OkHttp、HttpClient。
- RPC 调用：Dubbo、gRPC、Thrift、公司内部 RPC Client。
- 外部 SDK：支付、库存、用户、商品、风控、搜索等远程服务 SDK。

## 3. 不做生产 attach 的原因

当前安全约束下，不应把 agent 直接 attach 到生产 JVM。原因包括：

- 生产业务 JVM 不应承受额外字节码插桩风险。
- 插桩、类重转换、Advice 执行都会引入额外复杂度。
- 生产请求可能包含敏感数据，采集调用树、SQL、参数上下文存在合规风险。
- 生产问题的根因是“代码结构被生产规模放大”，不一定需要生产实测才能识别。

因此，CodePerf 不宣称“生产实测检测”，而是提供“上线前生产规模风险评估”。

## 4. 为什么从通用性能检测收敛

通用性能检测会同时覆盖慢 SQL、CPU 热点、内存分配、接口延迟、锁竞争、GC 等方向。这些方向都有价值，但第一阶段继续扩展会带来几个问题：

- 规则过宽，容易出现大量历史问题和噪声。
- 测试环境数据不一致时，动态结果不能代表生产。
- agent 能力容易被误解为生产采集能力，触碰安全边界。
- 团队真正想防的是已经发生过的“循环内 I/O 放大”事故。

收敛后，第一阶段只回答一个高价值问题：

```text
这次代码是否引入了会被生产数据量放大的循环内外部调用？
```

## 5. 现有能力的重新归类

现有能力不废弃，但需要重新定位：

| 当前能力 | 新定位 |
|---|---|
| 静态字节码扫描 | 第一阶段主引擎，用于发现循环内 I/O 结构 |
| `NPlusOneSuspect` | `LoopIoAmplificationRule` 的早期雏形 |
| 动态 attach agent | 测试/预发环境证据增强，不作为生产检测核心 |
| SQL/JDBC 采集 | 用于验证数据库类循环 I/O 是否在测试路径中发生 |
| HTML 报告 | 后续改造成风险评审报告 |
| `fail-on` 门禁 | 后续用于 CI 阻断高风险变更 |

## 6. 第一阶段成功标准

第一阶段不是证明“生产一定会慢”，而是证明：

- 代码结构存在循环内 I/O 风险。
- 该风险可能随生产数据规模线性放大。
- 测试环境未暴露不等于安全。
- 报告能给出明确位置、调用类型、置信度、生产规模依据和修复建议。
