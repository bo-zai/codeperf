# 路线图与评审清单

## 1. 阶段路线

### 阶段 0：文档评审

目标：

- 确认项目从“泛性能检测”收敛为“循环 I/O 放大风险检测”。
- 确认不做生产 attach。
- 确认生产规模画像只使用脱敏聚合指标。
- 确认第一阶段只聚焦循环内外部 I/O 风险。

产出：

- 本目录下评审文档。
- 评审结论。
- 是否进入编码阶段的决策。

### 阶段 1：规则重命名与规则设计落地

目标：

- 将当前 `NPlusOneSuspect` 的概念升级为 `LoopIoAmplificationRule`。
- 覆盖 DB、HTTP、RPC、SDK 等循环内外部 I/O。
- 输出 I/O 类型、置信度和修复建议。

不要求：

- 不要求生产采集。
- 不要求完整调用图。
- 不要求覆盖所有性能问题。

### 阶段 2：diff-aware scan

目标：

- 只关注本次变更引入或修改的方法。
- 降低历史存量问题噪声。
- 支持 PR 或 CI 门禁。

建议能力：

```text
scan-diff
-> 识别 git diff 涉及文件
-> 映射到类和方法
-> 只报告变更范围内的循环 I/O 风险
```

### 阶段 3：生产规模画像输入

目标：

- 支持 `perf-profile.yml`。
- 报告中展示生产 P95/P99 规模。
- 根据生产规模评估风险等级。

要求：

- 不连接生产数据库。
- 不存储生产明细。
- 标注画像来源和生成时间。

### 阶段 4：报告改造

目标：

- 报告用事故语言描述风险。
- 区分静态结构风险、测试/预发证据、生产规模推断风险。
- 给出修复建议。

报告不应只输出技术标签，例如 `N+1 Query`，而应解释为什么测试未暴露但生产可能出问题。

### 阶段 5：预发动态证据增强

目标：

- 保留 agent 作为测试/预发证据增强。
- 验证风险路径是否被执行。
- 采集测试环境下的实际调用次数、SQL 次数和调用树。

边界：

- 不进入生产环境。
- 不作为第一阶段主判断依据。

## 2. 评审清单

评审时应逐项确认：

- 是否认可项目新定位：上线前循环 I/O 放大风险检测。
- 是否认可不做生产 attach。
- 是否认可动态 agent 只用于测试/预发证据增强。
- 是否认可 `LoopIoAmplificationRule` 是静态主规则。
- 是否认可第一阶段不覆盖所有性能问题。
- 是否认可生产规模画像只使用脱敏聚合指标。
- 是否认可缺少生产规模画像时不能简单判定安全。
- 是否认可后续优先做 diff-aware scan，减少历史问题噪声。
- 是否认可报告需要输出修复建议和风险推断，而不是只输出技术标签。

## 3. 进入编码阶段的条件

满足以下条件后再继续编码：

- 项目定位已评审通过。
- 第一阶段范围已确认。
- `LoopIoAmplificationRule` 的 I/O 类型和置信度规则已确认。
- 生产规模画像的来源和安全边界已确认。
- CI/PR 门禁策略已确认。
- 报告输出文案和风险等级含义已确认。

## 4. 暂不进入第一阶段的事项

以下能力暂不作为第一阶段目标：

- 生产 JVM attach。
- 全量 APM 能力。
- CPU 火焰图。
- GC/锁分析。
- 内存泄漏分析。
- 所有慢 SQL 治理。
- 自动修复代码。

这些方向可以保留为后续扩展，但不应干扰第一阶段事故防护目标。
