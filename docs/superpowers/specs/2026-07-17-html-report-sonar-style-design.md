# CodePerf HTML 报告 Sonar 风格重设计方案

## 1. 背景

当前 `SourceScanHtmlReportWriter` 已经能生成本地 HTML 报告，包含概览、Top 风险、模块分组、文件分组、源码片段、调用链、归因和修复建议。但页面仍存在三个核心问题：

- 视觉上更像“卡片展示页”，不像企业代码扫描工具。
- 信息密度偏低，同一文件多个风险时需要大量滚动才能完成排查。
- 右侧目录虽然存在，但与问题列表、过滤和详情的联动不够清晰。

用户明确要求参考 Sonar/SonarQube 的展示与配色。本方案只确认 UI 与交互设计，不修改检测规则、JSON schema、CLI 扫描逻辑或上报逻辑。

## 2. Sonar 参考结论

公开文档体现出的 Sonar 设计原则：

- 问题以 issue 为核心，围绕规则、严重级别、代码位置、状态、作者、负责人和修复工作流组织。
- Severity 是重要的一等信息。新版文档中常见 `High / Medium / Low` 等影响级别，历史模式也保留 `Blocker / Critical / Major / Minor / Info`。
- Clean as You Code 强调只关注新增或变更代码，避免开发者为全部历史问题负责。
- issue flow、primary location、secondary location 用于解释问题路径，非常适合 CodePerf 的“循环位置 -> 调用链 -> I/O 行”展示。

参考资料：

- SonarQube Issues 文档：`https://docs.sonarsource.com/sonarqube-server/10.4/user-guide/issues`
- SonarQube Clean as You Code 文档：`https://docs.sonarsource.com/sonarqube-server/10.6/user-guide/clean-as-you-code`
- SonarQube 术语：`https://docs.sonarsource.com/sonarqube-cloud/appendices/glossary`

## 3. 目标

HTML 报告打开后，开发者应在 5 秒内回答四个问题：

- 这次扫描是否阻断。
- 阻断的是本次变更风险、历史风险，还是未归因风险。
- 需要优先修复哪些文件和行号。
- 每个问题为什么危险，以及应该怎么改。

报告不是管理大屏，不追求炫酷视觉；它是本地修复工作台，优先保证可读、可筛选、可定位、可快速跳转。

## 4. 总体布局

采用接近 Sonar Issues 页的“三栏诊断工作台”：

```text
┌──────────────────────────────────────────────────────────────┐
│ 顶部质量门禁摘要：PASS/FAIL、阻断数、总风险、扫描文件、模块数 │
├───────────────┬───────────────────────────────┬──────────────┤
│ 左侧过滤栏     │ 中间 Issue 列表                 │ 右侧详情/索引 │
│ 模块           │ 文件分组 + 紧凑问题行            │ 当前问题详情   │
│ 归因           │ 类名:行号 + 规则 + I/O + 置信度   │ 源码片段       │
│ I/O 类型       │ 默认优先展示本次变更与阻断问题    │ 调用链         │
│ 严重级别       │ 历史问题折叠或低优先级展示        │ 修复建议       │
└───────────────┴───────────────────────────────┴──────────────┘
```

窄屏场景下改为单列：顶部摘要、过滤器、问题列表、详情块顺序排列。报告必须离线可用，不引入外部 CSS、JS 或字体。

## 5. 顶部质量门禁摘要

顶部区域不再使用大面积渐变或大卡片，改为企业工具风格的浅色状态条：

- 左侧：项目名、扫描模式、报告生成时间。
- 中间：`质量门禁：失败/通过`，失败时使用红色小状态块，通过时使用绿色小状态块。
- 右侧：关键指标卡片，包含阻断风险、风险总数、扫描文件、模块数、解析错误。

颜色要求：

- 页面背景：`#f7f8fa`
- 主文字：`#172b4d`
- 辅助文字：`#5e6c84`
- 边框：`#dfe1e6`
- 主蓝：`#0b5cab`
- 失败红：`#d93025`
- 警告橙：`#f29900`
- 通过绿：`#1f883d`

这些颜色接近 Sonar/企业控制台风格，避免大面积高饱和色。

## 6. 左侧过滤栏

左侧过滤栏常驻展示，承载快速定位能力：

- 搜索框：支持文件名、类名、方法名、证据文本、提交人搜索。
- 模块过滤：`全部 / codeperf-demo-admin / codeperf-demo-app / codeperf-demo-common`。
- 归因过滤：`本次变更 / 历史风险 / 未归因`。
- I/O 类型过滤：`DB / Redis / MongoDB / GaussDB / HTTP / RPC / SDK`。
- 严重级别过滤：`WARN / INFO / CRITICAL`，保留现有数据模型命名。
- 置信度过滤：`HIGH / MEDIUM / LOW`。

过滤器更新时，中间 issue 列表与右侧索引同步隐藏不匹配项。空结果时显示“未匹配到风险”，而不是留下空白页面。

## 7. 中间 Issue 列表

中间区域是主工作区，不再以大详情卡片作为默认视图，而是以文件分组下的紧凑 issue 行为主。

文件分组展示：

- 文件名：只显示短文件名，例如 `AdminOrderAuditService.java`。
- 文件路径：以小字显示完整相对路径。
- 文件统计：风险数量、最高严重级别、主要 I/O 类型、是否包含本次变更风险。

Issue 行展示：

- 行号：例如 `L21`。
- 规则标题：`循环内外部 I/O 调用可能被生产数据量放大`。
- 标签：`LOOP_IO_AMPLIFICATION`、`WARN`、`HIGH`、`DB`、`本次变更/历史/未归因`。
- 方法名：例如 `auditDeliveries`。
- 证据摘要：例如 `orderMapper.selectById(orderId)`，单行省略。

点击 issue 行后：

- 行高亮。
- 右侧详情区展示该 issue 的完整诊断信息。
- URL hash 更新，方便复制链接和刷新保持定位。

## 8. 右侧详情与目录

右侧不再只是静态目录，而是“当前问题详情 + 快速索引”的组合。

当前问题详情包含：

- 标题：`AdminOrderAuditService.java:21`
- 规则：`LOOP_IO_AMPLIFICATION`
- 风险解释：说明循环内 I/O 在生产数据量下会造成 N 次外部访问。
- 源码片段：仿 GitHub/Sonar 风格，紧凑行高，命中行浅红背景，行号固定宽度。
- 调用链：`循环方法 -> 中间方法 -> I/O 方法`，直接触发时显示“直接在循环体内触发”。
- 循环范围：`loopStartLine-loopEndLine`、`loopCallLine`、`ioLine`。
- 归因信息：有提交人时展示姓名、邮箱、commit、时间；没有时展示“本次扫描未获得增量归因”。
- 修复建议：批量查询、循环外预加载、Map 回填、批量 RPC、缓存或合并 SDK 调用。

快速索引包含：

- 模块索引。
- 文件索引。
- 当前过滤结果中的 issue 索引。

点击索引时，中间列表滚动到对应位置，并同步选中右侧详情。

## 9. 信息优先级

默认排序：

1. 本次变更风险优先于历史风险，历史风险优先于未归因风险。
2. 严重级别高的优先。
3. 置信度高的优先。
4. DB、HTTP、RPC 优先于 SDK，SDK 优先于其他未知类型。
5. 同一文件内问题聚合展示，避免一个文件的问题散落在页面各处。

Top 风险区不再独立占据大面积区域。可以保留为顶部一行“优先处理”提示，但点击后仍进入中间 issue 列表，不再重复展示大卡片。

## 10. 交互边界

本地 HTML 允许少量原生 JavaScript，但必须满足：

- 不依赖网络资源。
- 不请求远程接口。
- 不读取额外本地文件。
- 只做过滤、选中、滚动、hash 定位和详情切换。

默认页面加载时：

- 若存在本次变更风险，自动选中第一条本次变更风险。
- 否则选中第一条阻断风险。
- 若没有风险，展示通过态和空状态说明。

## 11. 需要保留的现有能力

重构 UI 时必须保留：

- HTML 转义，源码和 evidence 中的 `<`、`>` 不能破坏页面。
- 源码文件不存在时报告仍生成成功。
- JSON 报告结构不变。
- `SourceScanHtmlReportWriter` 对外入口保持兼容。
- `ScanCommand` 控制台输出保持简洁，不输出 HTML 明细。
- Java 8 兼容，不使用 Java 9+ API。

## 12. 测试策略

新增或调整测试覆盖：

- 报告包含顶部质量门禁摘要。
- 报告包含左侧过滤栏、中间 issue 列表、右侧详情区。
- 同一文件多个风险时，中间列表展示多条 issue 行，右侧索引也展示多条 issue。
- 点击相关的 DOM 结构具备稳定 id 和 `data-*` 属性，便于 JS 选择。
- HTML 转义仍生效。
- 源码片段命中行高亮，行距紧凑。
- 缺失源码文件时显示“源码片段不可用”。
- 历史风险不被错误渲染为“本次变更风险”。

## 13. 不做事项

本轮不做：

- 不修改静态检测规则。
- 不修改 server 上报 payload。
- 不新增 `.codeperf.yml` 配置项。
- 不引入外部前端框架、CSS 框架或图标库。
- 不做管理层趋势大屏。
- 不做真实 Sonar 页面复制，只借鉴信息架构、配色和工作流。

## 14. 验收标准

方案实现后，应满足：

- 打开报告第一屏能看到质量门禁结果和核心指标。
- 一个多模块项目中，开发者能通过模块、文件、I/O 类型快速定位问题。
- 同一文件存在多个风险时，列表和详情数量一致，不再出现“页面显示 3 个、详情只有 1 个”的误解。
- 页面没有明显文字溢出，长路径和长 evidence 能省略或换行。
- 源码片段接近 GitHub/Sonar 的紧凑代码阅读体验。
- 视觉风格克制、清晰、企业化，不再像营销展示页。
