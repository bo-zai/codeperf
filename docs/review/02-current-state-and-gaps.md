# 当前能力与问题评估

## 1. 当前项目能力

CodePerf 当前已经具备一个可运行的 MVP：

- `scan`：基于 ASM 扫描 `.class`，发现静态性能嫌疑。
- `attach`：通过 Attach API 将 agent 挂入目标 JVM。
- agent：使用 ByteBuddy 插桩 Spring MVC、目标包方法和 JDBC。
- `report`：读取动态采集数据和静态扫描结果，生成 HTML 报告。
- `fail-on`：根据最高严重度返回退出码，可用于门禁。

这些能力对验证 demo 有价值，但与新的事故目标相比，需要重新调整优先级。

## 2. 当前静态扫描能力

当前静态规则包括：

- 循环内 Repository/DAO/Mapper/HTTP Client 调用，识别为 N+1 嫌疑。
- 循环内 `List.contains` 或嵌套循环，识别为 O(n²) 嫌疑。
- Math 调用密集或大整数常量，识别为 CPU 热点嫌疑。
- 循环内数组分配或大数组分配，识别为内存分配嫌疑。

其中，第一条与事故目标最相关。它应从 `NPlusOneSuspect` 升级为更通用的 `LoopIoAmplificationRule`。

## 3. 当前动态 agent 能力

agent 当前可以采集：

- HTTP 入口耗时。
- 方法调用树。
- JDBC SQL 指纹、次数、耗时。
- 请求线程栈采样。
- 单请求线程分配字节数。

这些数据适合在测试或预发环境补充证据，例如证明某个路径确实执行到了风险代码。

## 4. 关键问题

### 4.1 动态结果依赖测试数据规模

当前动态 N+1 判断依赖同一请求中 SQL 指纹的执行次数。测试环境数据少时，执行次数低，风险容易被低估。

示例：

```text
测试 SQL count = 5
生产 P95 loop count = 2000
```

当前规则无法从测试 count 推断生产风险。

### 4.2 不能宣称生产实测检测

由于安全约束，agent 不应进入生产环境。当前方案如果继续强调运行态画像，容易让评审误以为工具需要生产采集。

新方案必须明确：

```text
生产数据只提供脱敏规模画像，不采集生产运行时明细。
```

### 4.3 静态规则仍偏启发式

当前循环内 I/O 判断主要依赖类名和调用类型，例如 `Repository`、`Mapper`、`RestTemplate`、`Client`。这会带来误报和漏报。

可能误报：

- 循环次数固定很小。
- 调用的是本地内存方法，但类名包含 `Client`。
- 循环内调用已做缓存或批处理封装。

可能漏报：

- 循环体调用本地 Service，Service 内部才调用外部接口。
- 公司内部 RPC SDK 命名不包含常见关键字。
- 外部调用通过反射、动态代理或统一网关封装隐藏。

### 4.4 缺少生产规模画像

当前报告无法回答：

- 生产里这个循环通常会跑多少次？
- P95/P99 数据规模是多少？
- 测试环境和生产环境差距多大？
- 该风险是否足以阻断上线？

这是第一阶段必须补上的核心能力。

## 5. 当前能力保留策略

现有功能不删除，但重新归位：

- 静态扫描作为主判断入口。
- 动态 agent 作为测试/预发证据增强。
- 动态规则不再单独决定生产风险等级。
- CPU、内存、慢 SQL 作为后续扩展能力，不进入第一阶段主目标。
