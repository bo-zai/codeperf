# 目标方案

## 1. 总体思路

目标方案采用三层证据模型：

```text
静态结构风险
测试/预发运行证据
生产规模画像
```

工具最终输出的是“上线前生产规模风险评估”，不是“生产实测结论”。

## 2. 第一层：静态结构风险

静态规则是第一阶段主引擎。它不运行应用、不 attach JVM，只分析代码或字节码结构。

核心规则：

```text
LoopIoAmplificationRule
```

判断目标：

```text
方法中存在循环，且循环体内存在外部 I/O 调用。
```

这层回答：

- 哪个类、哪个方法存在风险？
- 循环体内调用了什么外部 I/O？
- I/O 类型是 DB、HTTP、RPC 还是 SDK？
- 静态置信度是多少？

## 3. 第二层：测试/预发运行证据

动态 agent 不作为生产检测核心，只用于测试或预发环境。

这层回答：

- 目标入口是否执行到了风险方法？
- 测试环境本次循环或 SQL 执行了多少次？
- 调用树中风险方法耗时是多少？
- 是否能提供实际 SQL 指纹、请求耗时、调用树证据？

这层不能单独证明生产安全，因为测试数据规模可能远小于生产。

## 4. 第三层：生产规模画像

生产规模画像只包含脱敏聚合指标，不包含业务明细。

这层回答：

- 生产 P50/P95/P99 循环规模是多少？
- 大租户、大客户、大部门场景规模是多少？
- 测试规模和生产规模的放大倍数是多少？
- 风险是否达到上线阻断标准？

示例：

```yaml
entries:
  "POST /api/orders/report":
    loopInputs:
      users:
        testP95: 5
        prodP95: 2000
        prodP99: 8000
```

## 5. 报告结论分类

报告必须明确区分三类结论：

| 结论类型 | 含义 |
|---|---|
| 静态结构风险 | 代码结构存在循环内 I/O |
| 测试/预发运行证据 | 该路径在测试或预发中被执行并采集到证据 |
| 生产规模推断风险 | 结合生产规模画像推断上线后可能放大 |

禁止把“测试环境未变慢”解释为“生产安全”。

## 6. 风险等级建议

第一阶段可采用以下评估逻辑：

| 条件 | 建议等级 |
|---|---|
| 循环内高置信 I/O，缺少生产规模画像 | WARN |
| 循环内高置信 I/O，生产 P95 规模较大 | CRITICAL |
| 循环内中低置信 I/O，生产规模未知 | INFO/WARN |
| 已证明循环次数固定且很小 | INFO 或忽略 |
| 已改为批量接口或预取 | 不报或降级 |

具体阈值应在评审后确定，避免在文档阶段过早固化。

## 7. 后续命令形态建议

后续实现可以保留现有命令，并新增能力：

```text
scan
-> 扫描全量或指定包，输出静态风险

scan-diff
-> 只扫描本次变更涉及的方法，适合 PR 门禁

report --profile
-> 合并静态结果、动态证据、生产规模画像
```

第一阶段优先实现 `scan-diff` 和 `--profile`，agent 能力保持辅助。
