# 05 · 静态源码分析引擎

> 关联：`00-overview-architecture.md`、`04-analysis-report.md`。本文档描述当前 CLI 静态扫描的落地方案。

## 1. 当前定位

静态分析是 CodePerf 第一阶段的主能力，运行在开发者本地 Git 流程中，用于在代码进入测试前发现循环 I/O 放大风险。

当前实现不再以字节码扫描作为主路径，而是基于源码 AST 分析：

- 可以直接扫描变更源码，不要求项目先完整编译。
- 可以报告具体源码文件、行号、循环范围和证据片段。
- 更适合 pre-commit、pre-push 和手工 `codeperf scan` 场景。

## 2. 扫描对象

CLI 从 `.codeperf.yml` 读取源码根，默认扫描 Git 变更文件；执行 `codeperf scan --all` 时扫描配置内全部源码。

多模块项目通过 `modules` 声明模块边界，报告按模块聚合展示。未匹配到模块的文件仍可扫描，但会归入默认模块或未识别模块，避免因为配置不完整而静默漏扫。

## 3. 核心规则：LoopIoAmplificationRule

`LoopIoAmplificationRule` 是静态主规则，用于识别“循环结构内触发外部 I/O”的风险。

检测步骤：

1. 识别 `for`、增强 `for`、`while`、`do while` 等循环结构。
2. 在循环体内查找直接 I/O 调用，例如 Repository、Mapper、DAO、RestTemplate、HTTP Client、RPC Client、SDK、Redis、MongoDB、GaussDB 等调用。
3. 在配置开启时，沿方法调用链向下追踪，发现“循环内调用本地方法，本地方法内部再触发 I/O”的间接风险。
4. 记录源码文件、方法名、循环起止行、循环内调用行、最终 I/O 行、I/O 类型、证据文本与调用链。
5. 结合 Git blame 做归因，标记本次变更风险、历史风险或未归因风险。

## 4. 风险分类

| 类型 | 示例 | 说明 |
|---|---|---|
| DB | `userMapper.selectById(id)`、`repository.findByUserId(userId)` | 可能形成数据库 N+1 或批量查询拆散 |
| Redis | `redisTemplate.opsForValue().get(key)` | 可能在生产大集合下形成大量缓存访问 |
| MongoDB | `mongoTemplate.find(...)` | 可能形成批量文档查询放大 |
| GaussDB | Mapper、DAO、Repository 调用 | 与 MySQL 类似，按数据库 I/O 处理 |
| HTTP/RPC | `restTemplate.exchange(...)`、`client.query(...)` | 可能放大外部服务调用次数 |
| SDK | 三方或内部 SDK 客户端调用 | 需要结合命名、包名与证据判断置信度 |

## 5. 输出与报告

控制台输出面向 Git 阶段快速决策：

- 扫描文件数、风险数、阻断数、门禁结果。
- 每个模块的风险数量。
- Top 阻断风险的规则、级别、文件、行号、方法与归因。
- JSON 与 HTML 报告路径，以及可直接在浏览器打开的 `file:///` 地址。

HTML 报告面向开发者修复：

- 顶部状态区展示通过或阻断。
- 模块区展示每个模块的扫描概览。
- 文件区展示每个文件的问题分布。
- 问题明细展示源码片段、调用链、证据、归因和修复建议。
- 右侧目录索引支持快速跳转到模块、文件和问题。

## 6. 与动态检测的关系

静态扫描负责提交前提前发现问题，动态 agent 负责在测试、预发或手工验证环境补充运行证据。两者不应混淆：

- 静态发现可以作为本地门禁。
- 动态证据只能说明某次运行触发了对应路径。
- 生产规模风险需要额外结合脱敏聚合指标或 APM 指标做推断。

## 7. 已知边界

- AST 分析无法证明某个循环在生产一定会遍历大量数据，只能证明结构上存在放大风险。
- 跨方法调用链存在深度限制，过深或动态分派复杂的调用可能漏报。
- SDK 类型识别依赖命名、包名和调用特征，可能存在误报，需要通过配置和证据逐步收敛。
- 对历史代码的归因依赖 Git blame；未提交、换行大规模调整或历史信息缺失时可能显示未归因。
